## Must-Fix (blocks ship)

### MF1. Fabricated client testimonials — 8 invented quotes across 3 pages

- `index.html:271-309` — three testimonials from "James M. (Logistics, Nelson)", "Sarah K. (Wellness Studio, Motueka)", "David P. (Professional Services, Richmond)"
- `portfolio.html:209-262` — four testimonials: James M., Sarah K., David P., "Lisa T. (Kiwi Craft Collective)"
- `about.html:232-247` — Sarah K. quote ("Working with Uriah feels like having a tech-savvy friend on your team…")

All quotes are invented. Replace with placeholder block ("Testimonials coming soon as projects launch") or remove sections until real ones exist.

### MF2. Fabricated portfolio case studies

`portfolio.html:124-202` lists six invented client projects with named businesses, project descriptions, and tag-lists across web design / branding / automation. Each is a fully-formed case study that doesn't correspond to any real engagement.

Options: (a) replace with placeholder ("Recent projects coming soon"); (b) replace with non-client work that *is* real (your toolkit, this site itself, audits-for-friends if any); (c) gate portfolio.html behind "coming soon" and remove it from main nav until Simons 2026 / InfoFocus 2026 ship.

### MF3. Fabricated stat counters

- `index.html:235` — `data-target="5"` "Happy Clients" (real value: 0 confirmed)
- `portfolio.html:91-108` — "100% Client Satisfaction / 5+ Projects Delivered / 3 Industries Served / 24hr Avg. Response Time"

The "Est. in NZ: 2024" counter on index is plausible (matches `about.html` "Est. July 2024"). The "100% Custom Builds" and "24hr Response Time" claims are aspirational but defensible. The headcount-style claims are not.

### MF4. Fabricated team / international claims

- `about.html:202-208` — "Multiple Industries — Logistics, wellness, professional services"
- `faq.html:212-216` — "We've worked with clients across Australia, the UK, the US, and beyond"
- `faq.html:413-416` — "Our team is 100% in-house or trusted long-term collaborators. All work is reviewed by ICU Media Design for quality and consistency" (implies plural team; you're solo)

Phrasing throughout uses "we" / "our team" — fine for a sole-trader brand voice but the *factual claims* of multiple industries / international clients / in-house team need to come down.

### MF5. Broken image references — `imgs/automation-leads.jpg`

Referenced by three pages but the file does not exist:
- `portfolio.html:191` — Kiwi Craft Collective card
- `solutions.html:206` — "Automated Lead Capture" overlay
- `blog.html:124` — "5 Automations Every Small Business Should Set Up Today"

Either add the asset or change the references to an existing image (`imgs/seo.jpg`, `imgs/agents.jpg`, etc.).

### MF6. Contact form is non-functional — Formspree placeholder

`js/script.js:800` posts to `https://formspree.io/f/YOUR_FORMSPREE_ID`. Every submission fails the `response.ok` check and shows the catch-block error message. Hard-block on receiving inquiries. ([H4-L3], recurring across all four harden passes.)

Options: real Formspree ID; switch to a Hostinger-friendly mail handler; or for v1 remove the form entirely and direct everyone to the `mailto:` link that's already in the contact page sidebar.

### MF7. Newsletter forms are non-functional — silent prevent-default

Seven instances of `<form ... onsubmit="event.preventDefault();">` (every page footer + `blog.html:208`). They show "Subscribe" but discard the email. Visitor expectation of subscription with zero delivery = bad first impression. Either wire to a real list (Mailchimp / Buttondown / a Formspree mailing-list endpoint) or remove the forms for v1.

### MF8. Production domain decision

Every `<link rel="canonical">`, `og:url`, JSON-LD `url`, and footer email points at `icumediadesign.com`. The handoff floated `icumediadesign.co.nz` as a possibility. Locking the domain affects:
- ~50 canonical/OG/JSON-LD references across 11 pages
- email setup (`hello@`, `urgent@`)
- DNS / SSL setup on Hostinger
- the JSON-LD `LocalBusiness` and `Organization` records

### MF9. Hostinger plan vs `server.js`

`server.js` is an Express app with helmet CSP, rate limiting, and three Stripe checkout endpoints (`/create-checkout-session`, `/create-subscription`, `/create-0-dollar-session`). It also references `success.html` and `cancel.html` which **don't exist** — the success/cancel flow is broken even on the dev side.

Standard Hostinger shared hosting is PHP/Apache static — `server.js` will not run there. The Stripe endpoints aren't linked from any HTML page (verified via grep), so they're effectively dead code as far as visitors are concerned.

Options: (a) deploy as static-only to shared Hostinger and treat `server.js` as legacy/local-dev only; (b) Hostinger VPS or Cloud plan to run Node; (c) rip Stripe out for v1 (it's not driving any current revenue) and wire it back in once you have an actual paid offering.

The CSP/inline-script issues from [H4-M1/M2] only fire if `server.js` is the prod host. If we go static, those become non-issues.

### MF10. Email accounts must actually exist

Every footer + contact page promises `hello@icumediadesign.com`. `faq.html:357` promises `urgent@icumediadesign.com` for emergency support. If either inbox doesn't exist or isn't checked, every contact path is broken regardless of forms working.

## Should-Fix (would-I-put-this-on-a-business-card)

### SF1. Working-hours claim vs reality

`contact.html:121-123` — "Mon–Fri, NZ business hours" + "We typically respond within 24 hours". Reality per `context/personal-info.md`: full-time landscaping job, evenings + weekends. Either soften the claim ("evenings and weekends, replies within 24-48 hours") or commit to a workable lunchtime check-in pattern. Honesty here protects against blown expectations once a real client lands.

### SF2. "Call us directly" with no number

`faq.html:220` — "Just reach out via our contact form, email, or call us directly". No phone number is published anywhere. Either add a number or remove "or call us directly".

### SF3. Hosting capability claims unverified

`faq.html:344-348` — "We offer reliable, New Zealand-based hosting options or can set you up on platforms like SiteGround, Cloudways, or your preferred provider. All hosting includes SSL certificates and daily backups." Confirm what hosting you actually deliver in v1 (resold from Hostinger? a CPanel reseller? "we'll point you at a host" advisory?). The claim is fine if backed; bad if not.

### SF4. Pricing inconsistencies

- E-commerce starting price: design.html FAQ says "$4,000+", design.html JSON-LD also "$4,000+", `faq.html:313` says "$4,500"
- GST treatment: design.html FAQ says "All prices exclude GST", `faq.html:325` says "15% GST is included for New Zealand-based clients" — directly contradictory
- Confirm every named price is one you'll honor (full list: $1,500 starter / $2,500–6k business / $4–4.5k+ ecommerce / $200/$400/$750 mo maintenance / $250/$450/$850 SEO audits / $200/$150/$350 mo analytics / $300 integrations / $500 automations / $500 logo)
- GST nuance: registration is only required at $60K NZD turnover; if not registered, no GST is charged at all and the include/exclude phrasing misleads

### SF5. `work.html` orphaned from main nav

Rich page (Discovery → Design & Plan → Build & Test → Launch & Train → Support & Grow timeline + "What Every Client Gets" 6-card grid + cross-links). Main nav goes to `portfolio.html`, never to `work.html`. Active-nav-link map in `js/script.js:694` knows about it (`'work.html': 'Portfolio'`) so the link existed at some point. Decide: link in nav, redirect to portfolio, merge content, or delete.

### SF6. Portfolio + Work overlap — strategic consolidation

`portfolio.html` = case studies (currently fictional). `work.html` = process / what-you-get (no fictional content, well-written). Strategically `work.html` could *replace* `portfolio.html` for v1 — process content is honest agency hygiene that doesn't need clients to back it. Worth considering as the path of least resistance to MF2.

### SF7. About-page founder photo

`imgs/uriah.jpg` (visible on `about.html`). At thumbnail size in the screenshot the photo reads as stylized / illustrated rather than a clean professional headshot. Verify it's actually you and not an earlier AI/avatar placeholder. If it's an avatar, replace with a real photo before launch — founders selling trust need a real face.

### SF8. Blog content is fabricated

`blog.html` ships with one featured post + six grid posts, all dated Dec 2025–Mar 2026, all with descriptive blurbs and "Read More →" links that go nowhere. None of the posts exist. Three options: (a) write the actual posts (deferred), (b) replace with "Coming soon" placeholder, (c) remove blog entirely from v1 (also remove from footer Company column on every page).

The launch-readiness plan already removed blog from main nav — blog is currently footer-only, which makes (c) the cheapest move.

### SF9. Mixed US / NZ-UK English

Inconsistent spellings: `design.html` mixes "optimization" (US) with surrounding NZ-UK content. `work.html` uses "Optimised" (UK/NZ). `faq.html` uses "mobile-optimised". For an NZ agency, standardize on NZ English (-ise / -our / -re) site-wide.

### SF10. Image filename hygiene

Mixed casing and a typo. Doesn't break anything but stops looking professional if anyone inspects element on a slow-loading image:
- `Craftsmanship.jpg` (only capitalised file in directory)
- `maintainance11.jpg` (typo: should be `maintenance11.jpg`)
- Numeric suffixes (`aesthetics6.jpg`, `design9.jpg`, `useability2.jpg`, `evolution6.jpg`, `seo1.jpg`, `analytics1.jpg`) — leftovers from earlier iteration cycles

### SF11. Counter animation timing

`initCounters()` in `js/script.js:734` animates from 0 every time the counter scrolls into view. If a connection is slow or the user scrolls past quickly the counter visibly counts up from 0. Consider rendering the target value as the initial textContent (so it's correct in the no-JS / pre-animation state) before the IntersectionObserver kicks in.

### SF12. Header parallax conflicts with shader

`initParallaxSections()` at `js/script.js:983` translates `.header-content` on scroll. The new shader background is on the same `.header` element. If the parallax pushes content out from under the shader at certain scroll positions you get a visible split. Spot-check on desktop + mobile, especially on long pages (design.html, faq.html).

### SF13. Dev comments in production HTML

Every page contains `<!-- FIXED Services dropdown -->` and similar dev breadcrumbs (`<!-- Core Services with submenu -->`, `<!-- Support Services with submenu -->` in solutions.html, `<!-- row -->` and `<!-- end of row -->` markers throughout design.html). Cosmetic but ships dev process artifacts to view-source.

### SF14. Footer copy implies blog is active

Every footer Company column links to `blog.html` ("Blog"). If you take the SF8 (c) path and remove blog, also strip the footer link site-wide.

### SF15. Inconsistent footer email anchor

Footer-bottom shows bare `mailto:hello@icumediadesign.com` link styled by default (`a` color). On dark theme it may read as un-styled blue-ish in some browsers. Spot-check styling.

## Nice-to-Have (defer post-launch)

### NH1. [H4-L4] Drop jQuery — ~87KB

`js/script.js:1010-1023` is the only direct jQuery use, and it targets `.search-toggle` / `.modal-toggle` / `.modalBox` — none of which exist in any HTML. **Dead code wrapping a 270KB+ vendor.** Removing both the dead jQuery block and the jQuery `<script>` includes is a clean win, but you also need to confirm Bootstrap 4.3.1 doesn't depend on jQuery internally for any of the dropdown affix / data-spy hooks the site actually uses (the `data-spy="affix"` and `data-spy="scroll"` on every `<body>` strongly suggests yes — verify before removing).

### NH2. Bootstrap mostly unused

Custom CSS handles most layout. Audit which Bootstrap classes actually render visible behaviour (`row`, `col-md-*`, `m-auto`, `flex-center`, `text-center`) versus dead utilities. Could potentially shed Bootstrap entirely with a small grid utility, but big change.

### NH3. Themify icons → SVG

`vendors/themify-icons` is a webfont (~70KB CSS + several font files). Used heavily but every icon used is enumerable. Migrating to inline SVG is smaller, sharper, and avoids the webfont FOUT. Defer.

### NH4. [H4-M3] `serve.mjs` decodeURIComponent crash

`serve.mjs:18` already wraps in try/catch with a 400 response — confirmed reading the file. **This [H4] item is already fixed in the uncommitted work.** Move from open to verified.

### NH5. [H4-M1/M2] CSP misconfiguration in `server.js`

If you stick with static Hostinger, `server.js` never runs in prod and these are non-issues. If you ever flip to running Express on Hostinger Cloud / VPS:
- Add `https://fonts.googleapis.com` to `styleSrc` and `https://fonts.gstatic.com` to `fontSrc`
- Either move the inline `<script>if(sessionStorage.getItem('isTransitioning')…)</script>` to an external file or add its SHA-256 hash (the existing hash on `server.js:27` doesn't appear to match the current inline-script content — verify before relying on it)

### NH6. GCP cleanup

`app.yaml`, `.gcloudignore` are leftovers from a Google App Engine experiment. If Hostinger is the only target going forward, delete or move to `legacy/`.

### NH7. Stripe endpoints are dead code

Three endpoints in `server.js:67-134` and corresponding env vars (`TEST_ONE_TIME_PRICE_ID`, `TEST_SUBSCRIPTION_PRICE_ID`). Nothing in any HTML page POSTs to them. Pair with NH6 — if `server.js` isn't deploying anywhere, these endpoints aren't visiting prod. Decide whether to keep for future or remove for tidiness.

### NH8. `success.html` / `cancel.html` referenced but missing

Pair of NH7 — Stripe endpoints redirect to these on success/cancel and they don't exist. Stub them or remove the references with the Stripe code.

### NH9. Image filename normalisation

Pair of SF10 if you also want to renumber (`aesthetics6.jpg` → `aesthetics.jpg`, etc.).

### NH10. Old screenshots dir

`screenshots/` mixes 23 prior dev-cycle screenshots with the 21 new ones from this audit. Either `.gitignore` the directory entirely (Puppeteer output is local-dev artefact) or curate.

### NH11. Dead jQuery handlers in script.js

Same as NH1 — `js/script.js:1010-1023` references `.search-toggle`, `.modal-toggle`, `.modalBox` selectors. None exist in HTML. Pure dead code regardless of jQuery decision.

### NH12. Unread `data-integration` attribute on contact form

`contact.html:126` — `<form id="contact-form" data-integration="email">`. No JS reads the attribute. Remove.

### NH13. Newsletter form duplicate styling on blog page

`blog.html:208` styles inline `style="max-width: 450px; margin: 0 auto;"` rather than using a class. Tiny but worth promoting to a CSS class if the form is kept (or moot if MF7 deletes them).

### NH14. portfolio + work consolidation

If SF6 doesn't go forward, eventual consolidation is still worth doing.

### NH15. favicon.ico missing

No favicon in repo or referenced in any `<head>`. Browser tabs show default icon. Quick win — generate from `imgs/enhancedlogo.png`.

### NH16. OG image is the logo PNG

`og:image` everywhere points at `enhancedlogo.png`. Designed social cards (1200×630) read better. Defer.

### NH17. [H4-L1] `isNavigating` defensive reset

Already fixed in uncommitted work — `js/script.js:505` now sets `isNavigating = false` before the same-page early return.

### NH18. [H4-L2] Escape-key null guard

Already fixed — `js/script.js:319-321` now has `if (closeBtn) closeBtn.click();`.
