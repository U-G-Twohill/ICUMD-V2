## Cross-cutting

### 1. 🔴 Hero shader — what's actually tunable, and how to make each page meaningfully different

**Important framing:** the shader *is* original code, just visually similar to the asset because the underlying GLSL pattern (flowing-lines plasma) is the same. The implementation is yours; the *look* it produces is whatever you tune. Think of it like rebuilding a guitar — same shape, your strings, your tuning.

**The 13 tunable parameters** (set per-page via `data-shader='{...}'` JSON on each page's `<header>`):

| Group | Param | Default | What it does |
|---|---|---|---|
| Colors | `bgColor1` | `[0.059, 0.075, 0.094, 1.0]` | Left-side gradient colour |
| | `bgColor2` | `[0.06, 0.14, 0.28, 1.0]` | Right-side gradient colour |
| | `lineColor` | `[0.0, 0.48, 1.0, 0.6]` | Flowing-line colour + alpha |
| Speed | `overallSpeed` | `0.2` | Master speed multiplier |
| Lines | `lineCount` | `16` (8 mobile) | Number of flowing lines |
| | `minLineWidth` | `0.02` | Thinnest line |
| | `maxLineWidth` | `0.2` | Thickest line |
| | `lineFrequency` | `0.2` | Wave oscillation rate |
| | `lineAmplitude` | `1.0` | Wave height |
| Warp | `warpFrequency` | `0.5` | Distortion tightness |
| | `warpAmplitude` | `1.0` | Distortion strength (0 = straight lines) |
| Spread | `offsetFrequency` | `0.5` | Horizontal positional variation |
| | `minOffsetSpread` | `0.6` | Min vertical line gap |
| | `maxOffsetSpread` | `2.0` | Max vertical line gap |
| Scale | `scale` | `5.0` | Zoom level (higher = more zoomed out) |

**You're already varying it per page** — but subtly. Current per-page settings:
- index: speed 0.2, count 16, alpha 0.6, scale 5
- about: speed 0.15, count 12, alpha 0.45, warp 1.2
- services: speed 0.18, count 14, alpha 0.5, scale 4.5
- design: speed 0.2, count 16, alpha 0.55, warp 0.8
- solutions: speed 0.25, count 18, alpha 0.5, warp 1.3 (fastest, most chaotic)
- portfolio: speed 0.22, count 14, alpha 0.5, warp 1.0
- faq: speed 0.12, count 10, alpha 0.35, scale 6 (slowest, sparsest)
- contact: speed 0.15, count 12, alpha 0.4, warp 0.9

**For more dramatic per-page differentiation, three axes to play with:**

a) **Hue** — currently every page uses the same blue (0, 0.48, 1.0). Could shift hue per page to give each its own personality. E.g.:
- Home: blue (current — keep as brand anchor)
- About: warmer teal — `[0.1, 0.55, 0.8, alpha]`
- Services: brand blue (parent — hub identity)
- Design: purple-shift — `[0.4, 0.3, 1.0, alpha]`
- Solutions: cyan-shift — `[0.0, 0.7, 0.9, alpha]`
- Portfolio: gold accent — `[1.0, 0.7, 0.2, alpha]`
- FAQ: muted blue (current dim)
- Contact: warm orange-red — `[1.0, 0.4, 0.3, alpha]` (action-oriented)

b) **Density character** — slow + sparse vs fast + dense conveys different vibes. Match to page intent:
- Home: balanced (current)
- About: slow + sparse + contemplative (current direction is right)
- Services: medium-fast + organised
- Design: medium + warpier (creative)
- Solutions: fast + dense + active (already the most aggressive — keep)
- Portfolio: medium + clean
- FAQ: very slow + sparse (already there)
- Contact: medium + warm + inviting

c) **Background gradient** — `bgColor1` and `bgColor2` haven't been varied at all yet. Subtle gradient shifts per page would land each one's mood without being loud. Current default is dark navy → slightly warmer dark navy. Could nudge each page's gradient endpoints.

**Beyond parameters — what could be added:**
1. **Click-through interaction** — perturb the shader where you click/tap (a `u_click` uniform that decays over a second). Adds a delightful "I touched it" moment on landing.
2. **Mouse-position parallax** — small effect where lines bend slightly toward cursor. Already feels alive, this would feel responsive. Disable on mobile.
3. **A second pattern entirely** — a fragment shader variant (e.g. orb-style noise, Voronoi cells, particle field) selectable per page via `data-shader-type="lines|orbs|noise"`. Heavier work but each page could feel genuinely distinct.
4. **Reactive accent on scroll** — line colour intensity drops as you scroll past the hero (fade-out integrated with scroll, not just `display: none`).

**Recommended next move on shader:** decide on hue per page first (lowest cost, biggest payoff), then experiment with density character, then evaluate whether interaction adds enough delight to justify the work. I can prototype a hue-shift variant on a single page (probably home or solutions) before committing to all 8.

### 2. 🟡 Mobile nav — sub-pages access for Services dropdown

**The bug:** on mobile, tapping "Web & Design" or "Solutions & Automations" in the dropdown opens its sub-sub-menu rather than navigating to the page. There's no one-tap path to design.html or solutions.html on mobile.

Looking at `js/script.js:408-425` — the mobile handler explicitly `preventDefault()`s clicks on `dropdown-submenu > .dropdown-item` and toggles a class instead.

**Three options, ranked:**

**(a) Add an "Overview" entry as the first sub-sub-item** — recommended.
```
Services ▼
  Web & Design ▶
    Overview      ← new, goes to design.html
    Design        (anchor to #design)
    Maintenance
    Growth Add-Ons
  Solutions & Automations ▶
    Overview      ← new, goes to solutions.html
    Our Process
    Our Solutions
```
Preserves the expand-on-tap behaviour, gives one-tap access to the page itself, no JS changes needed. Three nav items to add total.

**(b) Two-tap: first tap opens, second tap navigates** — feels clever but is a known anti-pattern (users don't know the rule and feel "stuck" on first tap).

**(c) Split: chevron icon expands sub-menu, label tap navigates** — cleanest UX but requires HTML restructure to introduce a separate expand control. Higher work, higher polish.

I'd start with (a) — fastest to ship and solves the problem cleanly. Revisit (c) later if mobile users complain about the extra depth.

### 3. 🔴 Section layout & density — research and per-page proposals

**General consensus on landing-page density** (synthesized; I haven't run a fresh literature search, but the patterns below have been stable for several years):

- **5-7 main sections** is the sweet spot for a small-business landing page. 8+ starts feeling "just keep scrolling".
- Sections should each carry **distinct intent** — repeating "Why us" energy across three sections is filler.
- **Visual rhythm** matters: don't put two card-grid sections back-to-back. Alternate cards / prose / stats / quote / CTA.
- **Contrast carries weight** — your light/dark alternation is a strong asset. The issue isn't the alternation, it's that on Home there are *too many sections* so the alternation flips every ~150px and reads as formulaic.
- **By-the-numbers / stats** work best as a thin band integrated with another section, not its own. 30s of horizontal space, not 30vh of vertical.
- **Process / how-we-work** lands hardest as a *visual* (timeline, infographic, or step-cards) not as text — and best on a contrast background that lets the visual breathe.

You're right that Home is the worst offender. Current order: hero → Why ICUMD → What We Do → How We Work → By the Numbers → Tools → Testimonials → CTA = **8 sections** + hero. Let me propose a tighter version below in the Home section.

### 4. 🟡 Hero treatment uniformity per page

**You're right that the same hero on every page costs above-the-fold space.** But the animation continuity is genuinely valuable — it's the brand thread.

**Three patterns possible:**

**(a) Full hero everywhere** — current. 100vh-ish header with title + subtitle. Costs ~600-800px of above-the-fold real estate on every inner page.

**(b) Compact hero on inner pages** — header drops to ~40vh (~300-400px). Title + one-line subtitle, animation continues. Saves ~half the above-the-fold cost while preserving the visual thread. **Recommended for non-landing pages.**

**(c) No hero, animation as section background** — used selectively (Contact in particular). Animation moves to be the background of a *content* section rather than its own slab. Requires generalising `shader-bg.js` to attach to any element with `data-shader`, not just `.header`.

**My proposal:**
- **Home / Services / About** — full hero (a). They earn it.
- **Design / Solutions / Portfolio / FAQ** — compact hero (b). Title + tagline only, ~35-40vh.
- **Contact** — no hero, animation behind the form section (c). Generalise shader-bg.js for this. (Solves Uriah's complaint about the form being squashed when shader was put behind it — proper sizing fixes that.)

Re: copy — most hero subtitles are fine but a couple can sharpen. Specifics in per-page sections below.

## Things you didn't mention but I think are worth flagging

### A. ⏸ Counter "0 → target" animation flicker (SF11)
Stats counters render `0` initially and animate up to the target value when scrolled into view. On a slow load or fast scroll, a viewer sees `0`s. **Easy fix:** render the target value as initial `textContent` so the no-JS / pre-animation state is correct; animation still works. About a 5-line change to `initCounters()`. Not blocking but cheap to add when we're touching the home page anyway.

### B. ⏸ 404 page
There's no `404.html`. Hostinger static will fall through to whatever default the host serves. `under-construction.html` could be repurposed as a friendly 404. Not urgent for v1 but worth thinking about.

### C. ⏸ Designed OG image (1200×630)
Currently the social card image for shares is the logo PNG. A designed card with site copy + visual reads better when shared. Defer; mention at launch-prep.

### F. ⏸ Performance audit on shader
