
   - Remove the `preventDefault()` on `.dropdown-submenu > .dropdown-item` clicks.
   - Add separate click handler on `.dropdown-chevron` that toggles the parent's `.active` class to expand the sub-menu.

## Chunk 5 — Shader performance fallback system

**Estimated:** 1 evening.
**Lives in:** `D:\Repos\Hero-Shader\` repo first; back-port to ICUMD when stable.

### Goal

Detect device capability and progressively reduce shader quality if FPS drops below an acceptable threshold. Currently the shader has a binary mobile/desktop split; a more granular runtime-feedback system would help low-end devices.

### Proposed approach

In `src/shader-bg.js`:

1. Track frame timing in the existing `render()` loop. Every 60 frames (~1s at 60fps), compute average FPS.
2. Define quality tiers:
   - **High** (default desktop): full lineCount, full DPR, full alpha, full warpAmplitude
   - **Medium**: lineCount −25%, DPR clamped to 1, alpha −10%
   - **Low**: lineCount −50%, scale +20% (zoomed out so fewer lines needed), warp halved
   - **Minimal**: lineCount = 4, glow disabled, circles disabled (radius 0, brightness 0)
3. If avg FPS < 45, drop one tier (rebuild shader with lower lineCount). If still < 45 after rebuild, drop another. Floor at Minimal.
4. Don't auto-upgrade — once dropped, stay there for the session. (Auto-upgrade leads to oscillation.)
5. Optional: respect a manual override via `localStorage.setItem('shader-quality', 'minimal')` for testing.

### Validation

- Test on a low-end Android phone (or Chrome DevTools' "low-end mobile" CPU throttle + 4× slowdown).
- Verify shader visibly reduces quality after ~2 seconds when FPS drops.
- Verify high-end devices stay on High.
- Verify the production `shader-bg.js` (constants version) handles tier-down via shader recompile, not via uniform changes.

- `D:\Repos\Hero-Shader\src\shader-bg.js` — perf detection + tier system
- `D:\Repos\Hero-Shader\src\shader-bg-playground.js` — same patterns (or skip — playground is for tuning, not perf)

### Chunk 5 — Shader perf fallback (in Hero-Shader repo)

**Implemented:** 2026-05-08
**Files changed (in `D:\Repos\Hero-Shader`):** `src/shader-bg.js`, `docs/parameters.md`. **NOT yet back-ported** to ICUMD's `js/shader-bg.js` per the plan's "after validation" gate.

**Summary**
- Refactored `src/shader-bg.js` to add an FPS-driven quality tier system. The previous structure kept all setup inline in `init()`; the new structure encapsulates `buildProgram(s)` and `setupForTier(tier)` so the shader can be re-compiled with reduced settings at runtime.
- Tiers and adjustments (each derived from the page's full `data-shader` settings):
  - **high** — defaults; DPR up to 2 on retina
  - **medium** — `lineCount` × 0.75, alpha × 0.9, DPR clamped to 1
  - **low** — `lineCount` × 0.5, `scale` × 1.2, `warpAmplitude` × 0.5, alpha × 0.85, DPR clamped to 1
  - **minimal** — `lineCount` = 4, `glowSpread` / `circleRadius` / `circleBrightness` = 0, `warpAmplitude` × 0.5, DPR = 1
- FPS tracking: a frame counter resets every 1000ms; if measured FPS < 45 the next step downgrades one tier and resets the counter. Floor is `minimal`. No auto-upgrade (avoids oscillation).
- LocalStorage override: `localStorage.setItem('shader-quality', 'minimal')` locks the tier and disables auto-downgrade. `localStorage.removeItem('shader-quality')` returns to auto behaviour. Documented in `docs/parameters.md`.
- Console log on each downgrade: `shader-bg: dropping quality from medium → low due to low FPS`.
- Did NOT touch `src/shader-bg-playground.js` — playground is for tuning, not perf, and full-quality rendering is the right behaviour there (per plan).
- Did NOT back-port to `D:\Websites\ICUMD Clone Designer\js\shader-bg.js`. Validation step (test on a low-end Android or DevTools 4× CPU throttle) should happen first.
