## Chunk 4 — Mobile nav (option c chevron-split)

**Estimated:** ~2 hours.

### Goal

On mobile, restore one-tap access to sub-pages (`design.html` / `solutions.html`) AND keep the parent `services.html` directly tappable. The current behavior preventDefaults clicks on parent items and toggles the sub-menu instead, leaving the parent page unreachable on mobile.

### Pattern (option c)

```
Services [chevron▶]            ← whole row navigates to services.html on tap of label
                                ← chevron is a separate tap target that expands sub-menu

  Web & Design [chevron▶]     ← whole row navigates to design.html
                                ← chevron expands further sub-sub-menu
    Design (#design)
    Maintenance (#maintenance)
    Growth Add-Ons (#evolution)

  Solutions & Automations [chevron▶]
    Our Process (#solutions)
    Our Solutions (#integrations)
```

### Implementation

1. **HTML:** Each `.dropdown-submenu > .dropdown-item` (and the top-level Services link) gets a sibling `<button class="dropdown-chevron" aria-label="Expand sub-menu">` element with a chevron icon. Existing structure of `<a>` immediately followed by `<ul class="dropdown-menu">` stays — the chevron sits between them visually but is a separate interactive element.
2. **CSS:** Layout the row as flex; the `<a>` takes most of the width and is fully tappable. The chevron button is ~44×44px (touch target) on the right. On desktop (>768px) the chevron is visually hidden (since hover is the trigger).
3. **JS (`js/script.js`):**
   - Remove the `preventDefault()` on `.dropdown-submenu > .dropdown-item` clicks.
   - Add separate click handler on `.dropdown-chevron` that toggles the parent's `.active` class to expand the sub-menu.
   - Top-level Services nav link: same pattern. Tap on "Services" navigates to services.html; tap on Services' chevron expands the sub-menu.

### Files affected by Chunk 4

- All HTML pages (nav block) — add chevron buttons. Same pattern across all 8 pages with footer-enhanced nav.
- `js/script.js` — `initMobileMenu()` rewrite
- `styles.css` — chevron styling, hide-on-desktop rules

### Edge cases

- Keyboard navigation: Tab through nav, Enter on link goes to page, Tab to chevron, Enter expands. Pattern should still work for keyboard users.
- Closing on outside-click: existing logic should still apply.
- `setActiveNavLink()` in script.js should still work since the parent link is still an `<a>` with the right text.

## Chunk 5 — Shader performance fallback system

**Estimated:** 1 evening.
**Lives in:** `D:\Repos\Hero-Shader\` repo first; back-port to ICUMD when stable.

### Goal

Detect device capability and progressively reduce shader quality if FPS drops below an acceptable threshold. Currently the shader has a binary mobile/desktop split; a more granular runtime-feedback system would help low-end devices.

### Proposed approach

In `src/shader-bg.js`:

1. Track frame timing in the existing `render()` loop. Every 60 frames (~1s at 60fps), compute average FPS.
2. Define quality tiers:
   - **High** (default desktop): full lineCount, full DPR, full alpha, full warpAmplitude
   - **Medium**: lineCount −25%, DPR clamped to 1, alpha −10%
   - **Low**: lineCount −50%, scale +20% (zoomed out so fewer lines needed), warp halved
   - **Minimal**: lineCount = 4, glow disabled, circles disabled (radius 0, brightness 0)
3. If avg FPS < 45, drop one tier (rebuild shader with lower lineCount). If still < 45 after rebuild, drop another. Floor at Minimal.
4. Don't auto-upgrade — once dropped, stay there for the session. (Auto-upgrade leads to oscillation.)
5. Optional: respect a manual override via `localStorage.setItem('shader-quality', 'minimal')` for testing.

### Validation

- Test on a low-end Android phone (or Chrome DevTools' "low-end mobile" CPU throttle + 4× slowdown).
- Verify shader visibly reduces quality after ~2 seconds when FPS drops.
- Verify high-end devices stay on High.
- Verify the production `shader-bg.js` (constants version) handles tier-down via shader recompile, not via uniform changes.
